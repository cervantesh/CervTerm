# Architecture Maturity Scoring Protocol v1

This protocol makes the CervTerm ≥8.0 goal reproducible.

## Canonical scope and authorities

Score every non-test, non-generated tracked `.go` file under `cmd/` and `internal/`, including Windows, Unix, GLFW and other build-tagged files whether or not they compile on the auditor's host. Tests, generated files, vendored code and build outputs are evidence but are not production structure scores.

The canonical manifest is generated by `go run ./scripts/capture-architecture-maturity-baseline.go`; it uses `git ls-files`, excludes `_test.go`, and excludes files whose first 2 KiB contain both `Code generated` and `DO NOT EDIT`. Baseline manifest: `docs/validation/architecture-maturity-phase0/scored-files.txt`, LF SHA-256 `fb3f0e052fe56eacd07d99e091879987e423a59f66815a2697313679e28c836c`.

The canonical original finding set is `docs/architecture-maturity/accepted-findings.md`, LF SHA-256 `ba4e12b929960dc753ab3b6546038b11753b1dc1c099f94c860d3e97b1b22c4a`. Newly accepted findings are appended to the final audit report and are equally binding.

## Canonical evidence commands

```text
go run ./scripts/capture-architecture-maturity-baseline.go -count 10
go run ./scripts/check-maturity-gates.go
go test ./...
go vet -unsafeptr=false ./...
go test -tags glfw ./internal/frontend/glfwgl ./cmd/cervterm -count=1
go test -race ./internal/mux ./internal/termimage ./internal/layoutstate -count=1
go run ./scripts/check-phase15-recovery.go -race
go test ./internal/vt -run=^$ -fuzz=<each named target> -fuzztime=60s
```

Windows-only production files require Windows CI plus the documented real-GUI matrix; Unix PTY files require Linux CI. `SKIP`, `UNRUN`, headless and cross-compile results are never substituted for required platform evidence. Auditors record Go version, GOOS/GOARCH, tags and unavailable commands.

## Independent procedure and acceptance

1. Two teams receive the same commit, manifest, finding set, commands and rubric. Round 1 is blind: no team sees the other's scores or prose.
2. Each team cites `file:line`, package-graph, tests, performance, ownership and domain-boundary evidence for every item.
3. Round 2 exchanges evidence and challenges. Each team independently publishes revised final scores; neither may negotiate a consensus score.
4. If teams disagree, the lower final item/dimension score controls acceptance. A factual dispute is resolved only by a new reproducible test/static query; without one, the lower score stands. There is no upward tie-break.
5. A dimension passes only when the lower final score is ≥8.0. No cross-team averaging, weighting, reconciliation upward or rounding up is permitted.
6. Each team's `Overall` is the arithmetic mean of its seven unrounded component totals. Acceptance uses that raw mean; display truncates to one decimal.
7. All 30 original findings and every newly accepted finding, including Low severity, must close. A failing row or open finding creates another one-by-one remediation slice and the full audit repeats on new current main.

## Item scale

Each dimension has five items scored exactly `0`, `1`, or `2`; the dimension is their sum (`0..10`). `2` requires the listed positive evidence across the full manifest, `1` means partial/bypassable/incompletely evidenced, and `0` applies when the listed hard-zero condition exists. One material counterexample defeats a `2`.

## Dimension-specific anchors

### Architecture maturity

| # | Item | 2 — enforced | 1 — partial | 0 / hard zero |
|---:|---|---|---|---|
| A1 | Current model | Architecture and ADRs match sampled production and name target-only invariants | Mostly current with bounded stale sections | Major subsystem/owner is materially misdescribed |
| A2 | Boundaries | Layer/import rules are documented and mechanically checked | Rules exist but rely on review or have exceptions | Cycle or forbidden lower-layer frontend/native dependency |
| A3 | Ownership | Every sampled resource has one owner, lifecycle and rollback | Ownership is inferable but not uniformly typed/tested | Shared mutable resource has ambiguous or double owner |
| A4 | Reversibility | Compatibility seam, atomic commits and rollback order are evidenced | Rollback exists but requires broad coupled revert | Architecture change is big-bang or irreversible without data loss |
| A5 | Operability | Detached diagnostics/metrics expose failures without values | Partial observability or manual-only evidence | Critical failure is silent or leaks sensitive values |

### Clean Code

| # | Item | 2 — enforced | 1 — partial | 0 / hard zero |
|---:|---|---|---|---|
| C1 | Single reason to change | Large types/files are cohesive or have tracked bounded exceptions | Mixed responsibilities remain but have active split seams | Untracked god object/file coordinates unrelated domains |
| C2 | Intent/typing | Names and closed types/catalogs replace magic strings | Mixed typed/string APIs with bounded adapters | New core path depends on unvalidated string/`any` vocabulary |
| C3 | Local flow | Errors, state transitions and side effects are explicit/local | Some long branching or remote mutation remains | Normal control flow depends on panic, hidden global or silent fallback |
| C4 | Duplication/dead code | Authorities are singular; shims have expiry and tests | Duplication/shims remain tracked | Competing authorities or reachable dead compatibility surface has no owner |
| C5 | Change locality | Representative changes touch one authority and bounded consumers | Some shotgun edits remain | One closed-value change requires independent edits in ≥4 authorities |

### GRASP responsibility assignment

| # | Item | 2 — enforced | 1 — partial | 0 / hard zero |
|---:|---|---|---|---|
| G1 | Information Expert | Decisions live with owned data and invariants | Some projection/controller decides remote state | Repeated remote field mutation bypasses owner API |
| G2 | Controller | Each controller coordinates one named use case | Controller still spans adjacent concerns | Facade/controller directly coordinates ≥4 unrelated subsystems |
| G3 | High cohesion | Types and packages have coherent change causes | Bounded mixed helpers remain | Service locator or generic callback bag is a central dependency |
| G4 | Low coupling | Consumer-owned narrow ports and detached values dominate | Concrete dependencies remain behind adapters | New controller retains `*App`/`*Mux` or facade-equivalent interface |
| G5 | Protected Variations | Interfaces correspond to real variation and ≥2 consumers | Valid seam has one temporary consumer and expiry | Broad speculative interface/`any` erases domain result types |

### Dependency graph hygiene

| # | Item | 2 — enforced | 1 — partial | 0 / hard zero |
|---:|---|---|---|---|
| D1 | Acyclic graph | Canonical graph reports zero SCC cycles | N/A | Any production package cycle |
| D2 | Layer direction | All imports satisfy architecture allowlists | Reviewed exceptions are explicit and expiring | Core/VT/render/mux/config imports frontend/native projection |
| D3 | Stability direction | Volatile adapters depend on stable domain contracts | A few stable packages depend on volatile helpers | Stable domain package depends on GLFW/OpenGL/platform implementation |
| D4 | Surface minimization | Exported APIs and ports are narrow/non-bypassable | Some broad legacy adapters remain | Generic service locator or concrete facade escape bypasses owner |
| D5 | Automated enforcement | Graph, generated parity and forbidden edges run in CI | Checks exist but omit one material edge/vocabulary | Known forbidden dependency can merge without a failing gate |

### Domain isolation

| # | Item | 2 — enforced | 1 — partial | 0 / hard zero |
|---:|---|---|---|---|
| I1 | UI isolation | Core/VT/render/mux/config contain no GLFW/OpenGL/native UI | Build-tag exception is typed and documented | Native/UI handle crosses into a domain package |
| I2 | Domain model purity | Parser/core models carry bounded domain values only | Projection concern remains in a compatibility DTO | Core cell/parser state owns GL/native/external-effect behavior |
| I3 | Typed transfer | Cross-domain data is detached, typed and generation-addressed | Some map/string adapter remains bounded | Shared mutable pointer crosses owner/domain boundary |
| I4 | Effect isolation | External effects require policy-owned validated adapter | One effect path is manual/review-gated only | Parser/core directly opens URI, process, file or notification |
| I5 | Context identity | Window/workspace/tab/pane context is explicit end-to-end | Some legacy active/global lookup remains with rejection tests | Silent singleton/global routing can mutate the wrong window/pane |

### Ownership and transactions

| # | Item | 2 — enforced | 1 — partial | 0 / hard zero |
|---:|---|---|---|---|
| O1 | Singular owner | Every sampled durable/runtime resource names one owner | Ownership is documented but one adapter can retain it | Double owner, leaked owner or unowned durable resource |
| O2 | Atomic publication | Prepare/commit/rollback is old-or-new with fault matrix | Transaction exists but not every fallible edge is injected | Partial sibling/config/model publication is observable |
| O3 | Lifecycle | Close/cancel is idempotent and reverse-order | Lifecycle is correct but manual-only on one path | Double close/use-after-close/leak on tested path |
| O4 | Concurrency/time | Owner thread, clock and queue ownership are typed/race-tested | Implicit clock/thread remains behind serialization | Worker mutates owner state or erased result/clock crosses queue |
| O5 | Stale/failure evidence | Wrong-owner, stale generation and shutdown races fail without mutation | Some cases tested, coverage incomplete | Known stale/wrong-owner request mutates silently |

### Test and guardrail maturity

| # | Item | 2 — enforced | 1 — partial | 0 / hard zero |
|---:|---|---|---|---|
| T1 | Behavior invariants | Critical current and corrected behavior has tests; known defects expire | Characterization exists but gaps remain | Movement proceeds without characterization or defect golden becomes permanent |
| T2 | Adversarial gates | Race, fuzz, recovery and hostile fragmentation cover boundaries | Some gates are short/manual | Required boundary has no adversarial test or active crash corpus |
| T3 | Performance | Canonical ten-sample/3% and allocation protocol is reproducible | Measurements exist but environment/artifact incomplete | Unexplained >3% median or any allocation regression is accepted |
| T4 | Platform evidence | Windows/Linux/real-GUI evidence is honestly classified | One required platform remains UNRUN and blocks claim | SKIP/headless/cross-compile is labeled PASS |
| T5 | CI/static guardrails | CI enforces graph, ownership, catalog, redaction and generated parity | One enforcement remains review-only with tracked slice | Material guardrail can regress undetected |

## Required final report

Record commit, platforms/toolchains/tags, this protocol's canonical LF content SHA-256 (`git show <commit>:docs/architecture-scoring-protocol.md | sha256sum`), manifest path/hash, finding-set path/hash, every command/result, both blind Round-1 reports, Round-2 challenges, both final item and dimension scores, lower-team acceptance table, finding closure table, package graph, performance comparison and links to durable evidence.
